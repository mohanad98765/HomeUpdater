# DECISIONS.md
## سجل القرارات المعمارية (Architectural Decision Records)

> كل قرار تقني مهم يُسجّل هنا مع **السبب**. عندما نرجع للمشروع بعد فترة، هذه الوثيقة تذكّرنا بـ "لماذا" قررنا كل شيء.

**صيغة كل قرار:**
- **العنوان:** ما هو القرار
- **الحالة:** مقترح / معتمد / مرفوض / مُلغى
- **السياق:** الوضع الذي دعا للقرار
- **القرار:** ما الذي قررناه
- **النتائج:** ما الذي يؤثر عليه هذا القرار
- **البدائل التي درسناها:** الخيارات الأخرى ولماذا رفضناها

## ADR-001: استخدام Python للـ Backend
- **التاريخ:** 2026-04-27
- **الحالة:** ✅ معتمد
- **السياق:** نحتاج لغة فيها مكتبات قوية للشبكات وأمن المعلومات وتفاعل مع SSH/ADB/HTTP.
- **القرار:** Python 3.11+ مع FastAPI.
- **النتائج:**
  - مكتبات جاهزة: scapy, paramiko, adb-shell, python-nmap
  - FastAPI سريع ويولّد Swagger تلقائيًا
  - أداء كافٍ للشبكة المنزلية (<100 جهاز)
- **البدائل:**
  - Node.js: مكتبات أقل في مجال الشبكات الحقيقية
  - Go: أسرع لكن أقل مكتبات للـ scrapers والـ ADB
  - Rust: مبالغ فيه لمشروع منزلي

## ADR-002: SQLite كقاعدة بيانات في v1
- **التاريخ:** 2026-04-27
- **الحالة:** ✅ معتمد
- **السياق:** نحتاج تخزين دائم للأجهزة والتحديثات والتاريخ.
- **القرار:** SQLite (ملف واحد).
- **النتائج:**

  - سهل النسخ الاحتياطي (نسخ ملف واحد)
  - أداء كافٍ لشبكة منزلية
- **البدائل:**
  - PostgreSQL: زيادة تعقيد بدون فائدة لشبكة منزلية
  - JSON files: لا يحتمل العمليات المتزامنة

## ADR-003: واجهة Web بدلًا من Native Desktop App
- **التاريخ:** 2026-04-27
- **الحالة:** ✅ معتمد
- **السياق:** المستخدم يحتاج واجهة سهلة، ولكن يفضّل أيضًا الوصول من جواله أحيانًا.
- **القرار:** Web UI (React + Vite) محلية على `localhost:8080`، ويمكن الوصول من نفس الشبكة.
- **النتائج:**
  - تطوير أسرع
  - يعمل من أي جهاز في الشبكة (موبايل، تابلت، PC)
  - لا حاجة لتثبيت تطبيق على كل جهاز
- **البدائل:**
  - Electron: حجم كبير + تعقيد إضافي
  - Native (WPF / WinUI): يربطنا بـ Windows فقط
  - CLI فقط: غير مناسب للمستخدم العادي

## ADR-004: التشغيل على Windows PC في v1 (لا Docker إلزامي)
- **التاريخ:** 2026-04-27
- **الحالة:** ✅ معتمد
- **السياق:** المستخدم اختار Windows PC.
- **القرار:** نوفّر `.exe` واحد (PyInstaller) يثبّت كـ Windows Service. Docker Compose اختياري للمتقدمين.
- **النتائج:**
  - تجربة مستخدم نقرة-واحدة
  - لا حاجة لمعرفة Docker
  - يشتغل تلقائيًا عند تشغيل الكمبيوتر
- **البدائل:**
  - Docker إلزامي: عبء على المستخدم العادي
  - Manual Python install: غير مناسب لغير المبرمجين

## ADR-007: لا نخزّن credentials بنص واضح
- **التاريخ:** 2026-04-27
- **الحالة:** ✅ معتمد
- **السياق:** نحتاج كلمات سر/مفاتيح للوصول لأجهزة (راوتر، SSH، إلخ).
- **القرار:** كل الـ credentials مشفّرة بـ AES-256 بمفتاح مشتق من passphrase المستخدم (Argon2id).
- **النتائج:**
  - عند فتح البرنامج، يطلب passphrase مرة واحدة
  - لو تسرّب الـ DB، الـ credentials محمية
- **البدائل:**
  - keyring system OS: مرتبط بنظام تشغيل واحد
  - بدون تشفير: غير مقبول أمنيًا

## ADR-008: لا نطبّق تحديث Firmware تلقائيًا بدون تأكيد المستخدم
- **التاريخ:** 2026-04-27
- **الحالة:** ✅ معتمد (يخالف اختيار المستخدم بـ "تلقائي بالكامل")
- **السياق:** المستخدم اختار "تلقائي بالكامل"، لكن firmware updates يمكن أن تخرّب الجهاز إذا فشلت.
- **القرار:** نسوّي تفريقًا بين:
  - **Software updates** (تطبيقات، حزم نظام): تلقائي بالكامل ✅
  - **Firmware updates** (راوتر، تلفزيون، IoT): يحتاج موافقة صريحة لكل عملية ⚠️
- **النتائج:**
  - تجربة قريبة من "تلقائي" مع حماية من كارثة Bricking
  - المستخدم يقدر يفعّل auto-firmware يدويًا في الإعدادات على مسؤوليته
- **البدائل:**
  - تلقائي 100%: خطر ماسح للأجهزة
  - تأكيد لكل شيء: تجربة سيئة، ضد طلب المستخدم

> **هذا قرار حرج. ناقشه مع المستخدم في أول جلسة قادمة.**

## ADR-011: التسليم كـ Installer واحد (.exe / .msi) مع كل الاعتمادات مضمّنة
- **التاريخ:** 2026-04-27
- **الحالة:** ✅ معتمد (طلب صريح من المستخدم)
- **السياق:** المستخدم طلب أن يكون البرنامج "installable" وأن **كل البرامج المطلوبة (Python, Nmap, Node, إلخ) تُثبَّت تلقائيًا معه** بدون أي تدخل يدوي.
- **القرار:** بناء Installer شامل (Bundled Installer) باستخدام **Inno Setup** (مجاني، مفتوح المصدر، احترافي).
  - الـ Installer يحتوي على:
    - **Python embedded** (نسخة مضغوطة، لا تتطلب تثبيت Python على النظام)
    - **Nmap** (مضمَّن داخل الـ installer ويُستخرج عند التثبيت)
    - **Node.js portable** (للواجهة فقط في وقت التطوير، الإنتاج يُجمَع كـ static files فلا حاجة)
    - **Backend code** (مُجمَّع بـ PyInstaller)
    - **Frontend bundle** (React build جاهز كـ static)
    - **SQLite** (مدمج مع Python)
- **النتائج:**
  - المستخدم يضغط مرتين على ملف `HomeUpdater-Setup.exe`، يقبل الشروط، ويبدأ البرنامج
  - **لا يحتاج تثبيت Python أو Nmap أو Node بنفسه**
  - حجم الـ installer سيكون ~150-250 MB (مقبول)
  - يدعم: Start Menu shortcut, Desktop shortcut, Auto-start service, Uninstaller
- **البدائل التي درسناها:**
  - **NSIS:** أصعب في التخصيص، واجهة أقل أناقة
  - **WiX Toolset (.msi):** احترافي لكن صعب على المطوّر
  - **Electron Builder:** لو الـ Frontend Electron — لكن قررنا Web UI
  - **MSIX:** متطلبات Windows 10+ ومحدوديات Microsoft Store

> **هذا قرار حرج. Inno Setup مع Python embedded + PyInstaller + Nmap bundled هو الحل الأفضل.**

## ADR-012: الهوية البصرية والتجربة البصرية (Visual Identity & UX)
- **التاريخ:** 2026-04-27
- **الحالة:** ✅ معتمد (طلب صريح من المستخدم)
- **السياق:** المستخدم طلب "الرؤية البصرية والشكل الرائع" — البرنامج لازم يكون احترافيًا وجذابًا.
- **القرار:** نضع معايير تصميم صارمة في ملف `BRAND.md`:
  - **اسم تجاري مقترح:** **HomeUpdater** أو **DarUpdate** (نناقش الاسم النهائي)
  - **شعار:** أيقونة منزل مع سهم تحديث دائري — ألوان متدرّجة
  - **لوحة الألوان:**
    - Primary: أزرق عميق `#0D47A1` (يعطي إحساس الأمان والثقة)
    - Accent: أخضر-تركواز `#26A69A` (للحالات الصحية)
    - Warning: برتقالي `#FF9800`
    - Danger: أحمر `#D32F2F`
    - Background: أبيض/رمادي فاتح في الوضع الفاتح، رمادي داكن في الوضع المظلم
  - **خطوط:**
    - عربي: **Cairo** أو **Tajawal** (Google Fonts، مجانية)
    - إنجليزي: **Inter** أو **Plus Jakarta Sans**
  - **مكتبة UI:** shadcn/ui (احترافية، حديثة، قابلة للتخصيص)
  - **الرسوم المتحركة:** Framer Motion (للانتقالات السلسة)
  - **الأيقونات:** lucide-react (متّسقة، حديثة)
  - **الوضع المظلم/الفاتح:** كلاهما مدعوم
  - **RTL أولًا:** كل التصميم مبني للعربية أولًا
- **النتائج:**
  - تجربة بصرية على مستوى منتجات تجارية (مثل Notion, Linear, Vercel)
  - شعور بالاحترافية يمكن أن يجعل المشروع قابلًا للنشر تجاريًا
- **البدائل المرفوضة:**
  - Bootstrap: قديم
  - Material-UI: ثقيل ومرتبط بـ Google
  - Bulma: محدود

## ADR-013: تثبيت قرار Stack بعد مراجعة بدائل Windows-native
- **التاريخ:** 2026-04-27
- **الحالة:** ✅ معتمد (مراجعة بعد مدخلات المستخدم حول .NET/WPF/MSIX)
- **السياق:** المستخدم شارك معرفة شاملة عن خيارات Windows: .NET (WinForms/WPF/WinUI 3)، C++ Native، Cross-platform (Electron/Qt/MAUI). راجعنا القرار.
- **القرار:** **نستمر مع Python (Backend) + React (Frontend) + Inno Setup (Installer)** ونرفض التحوّل لـ C# WPF/WinUI، وذلك لأن:

  1. **مكتبات فحص الشبكة ضرورية:** scapy, paramiko, adb-shell, zeroconf, python-nmap موجودة في Python فقط بمستوى نضج عالٍ. C# يفتقد الكثير منها.
  2. **سرعة التطوير أولوية المستخدم:** React + shadcn/ui + Tailwind = أسرع UI في 2026.
  3. **AI assistance لـ Python+JS أوفر بكثير من C# WPF**، مهم للمستخدم غير المبرمج.
  4. **مرونة النشر المستقبلي:** نفس الكود يعمل على Linux/Mac إذا أردنا لاحقًا.
  5. **التغليف ممكن:** PyInstaller + Inno Setup + Python embedded يعطينا تجربة Windows أصيلة.

- **إضافات لتعزيز "Windows-native feel":**
  - [ ] Tray icon (pystray)
  - [ ] Windows Toast Notifications (winrt)
  - [ ] Auto-start Service
  - [ ] Splash screen عند الفتح
  - [ ] أيقونات نظام (taskbar, start menu)
  - [ ] دعم Jump Lists على taskbar (اختياري لاحقًا)

- **النتائج:**
  - الإصدار الأول: v1 خلال 8-12 أسبوع تقريبًا
  - الـ Installer النهائي: `HomeUpdater-Setup.exe` بحجم ~200 MB
  - تجربة لا تختلف عن أي تطبيق Windows عادي

- **البدائل المرفوضة:**
  - **C# WPF بالكامل:** نخسر مكتبات الشبكة الناضجة، نضاعف وقت التطوير
  - **Hybrid (C# UI + Python service):** تعقيد الصيانة لـ stack مزدوج، غير مبرر
  - **Tauri (Rust+Web):** خفيف لكن منحنى التعلم منخفض الدعم بالعربية
  - **Electron:** ثقيل بلا داعٍ (~150 MB لكل تطبيق)

> **هذا القرار يلغي إمكانية إعادة النظر إلا عند ظهور blocker حقيقي.**

## ADR-015: التنزيل المباشر (Direct Download) بدلاً من winget للـ Setup
- **التاريخ:** 2026-04-27
- **الحالة:** ✅ معتمد (بعد تجربة فاشلة لـ winget)
- **السياق:** على نظام المستخدم (Windows 11 build 26100)، رفض winget الاستجابة لأمر `winget --version` رغم أنَّ Windows 11 يَفترض أن يُتيحه. هذا قد يحصل في إصدارات N/KN أو إذا لم يكن Microsoft Store مُفعَّلاً.
- **القرار:** يَعتمد `setup.ps1` الآن على **تنزيل installers مباشرة** من المصادر الرسمية:
  - Python: `python.org/ftp/python/...`
  - Node.js: `nodejs.org/dist/...`
  - Nmap: `nmap.org/dist/...`
  - Git: `github.com/git-for-windows/git/releases/...`
  
  ثم يُشغِّل كلَّ installer بأعلام silent install ويتحقَّق من نجاح التثبيت.
- **النتائج:**
  - يعمل على أيِّ نظام Windows 10/11 بصرف النظر عن وجود winget.
  - الإصدارات مُثبَّتة (pinned) ومعروفة — لا مفاجآت من تحديث winget.
  - حجم الـ caching يصل لـ ~150MB في `%TEMP%\HomeUpdater-Installers\` (مفيد لإعادة المحاولة).
  - Installer النهائي (Inno Setup في Phase 4) لن يحتاج هذه الخطوات أصلاً لأنَّه سيَضمن كل شيء داخله.
- **البدائل المرفوضة:**
  - winget كأولى: متعطِّل على بعض الأنظمة كما رأينا.
  - Chocolatey: يحتاج تثبيتاً مسبقاً، نفس مشكلة winget.
  - Python Embedded + Node Portable: مفيد للـ Installer النهائي، لكن في طور التطوير نريد تثبيتاً كاملاً.
